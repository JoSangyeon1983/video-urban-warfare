# LLM Wiki — Schema & Operating Manual

Andrej Karpathy의 **LLM Wiki** 패턴을 따르는 개인 지식 저장소.
원본을 질문할 때마다 다시 읽는 RAG 방식 대신, **수집(ingest) 시점에 지식을 위키로
컴파일**하고 시간이 지날수록 복리(compounding)로 축적함. LLM(Claude)이 마크다운
문서를 직접 작성·편집·연결하는 **위키 에디터** 역할을 함.

이 파일은 LLM이 위키를 **수집/유지/질의**하는 방법을 정의하는 스키마임.
새 세션을 시작할 때 가장 먼저 이 파일을 읽음.

---

## 1. 핵심 원칙

1. **3-레이어 분리** — 불변 원본(`sources/`) · LLM 소유 위키(PARA 폴더) · 스키마(이 파일).
2. **LLM이 위키 에디터** — 사용자가 주제/소스를 던지면 Claude가 문서를 생성·갱신함.
3. **복리 축적** — 한 번에 완벽히 쓰지 않고, 수집·질의를 거듭하며 보강함.
4. **출처 추적(provenance)** — 모든 주장은 `sources/`의 source-id로 역추적 가능해야 함.
5. **Atomic Notes** — 한 파일에 한 주제. 파일명이 곧 제목.
6. **엔티티 1급** — 고유명사(사람·조직·모델·제품·논문)는 `type: entity` 페이지로 분리해 상호연결의 허브로 삼음.
7. **링크 하한선** — 모든 페이지는 관련 페이지를 **최소 2개** `[[ ]]`로 연결함(고아 방지·복리 그래프 강제).
8. **한국어 우선** — 문서는 한국어로 작성. 기술 용어는 영문 병기 가능.
9. **교차검증(cross-model review)** — 파급 큰 문서는 작성한 모델이 아닌 **독립 모델(Codex)**이 검수함. 자기 글 자기 검수는 환각을 못 잡으므로 작성자(Claude)와 검수자(Codex)를 분리함. §7 참고.

## 2. 3-레이어 아키텍처

| 레이어 | 위치 | 소유자 | 변경 규칙 |
|--------|------|--------|-----------|
| **Raw sources** | `sources/` | 사람 | 불변(immutable). LLM은 **읽기만** 함. |
| **Wiki** | PARA 6폴더 (`00_~05_`) | LLM | LLM이 생성·갱신·연결함. |
| **Schema** | `CLAUDE.md` (이 파일) | 사람+LLM | 규약을 정의함. |

> `sources/`(원본·불변)와 `03_References/`(LLM이 쓴 요약·인용 페이지)는 다름.
> 원본은 `sources/`에 그대로 보존하고, 그 가공물만 위키 레이어에 둠.

## 3. 디렉토리 구조

```
/                     # vault root
├── CLAUDE.md         # 이 파일 — 스키마/작업 지침
├── sources/          # ★ 원본(불변) — 논문·기사·데이터·메모 + source-id 레지스트리
│   └── README.md     # 소스 등록 규칙, source-id 인덱스 표
├── 00_Index/         # 목차·지도 문서 (MOC: Map of Content) + answer 환원 페이지 등록
├── 01_Concepts/      # 개념 정리 (정의·원리·이론) + 엔티티(type:entity) + 종합(type:overview)
├── 02_HowTo/         # 실용 가이드, 레시피, 튜토리얼
├── 03_References/    # 논문 요약, 인용, 외부 링크 정리 (sources/의 가공물)
├── 04_Projects/      # 프로젝트별 노트
├── 05_Logs/          # log.md (append-only) + 학습 일지·실험 기록
└── assets/           # 이미지, 첨부파일
```

## 4. 문서 작성 규칙

### 파일명
- 공백 허용 (Obsidian 호환). 한국어 파일명 가능 (예: `트랜스포머.md`).
- 영문 약어가 보편적이면 영문 사용 (예: `LLM.md`, `RAG.md`).

### 프론트매터 (YAML)
모든 문서 상단에 포함:

```yaml
---
title: 문서 제목
type: entity | concept | overview | howto | reference | project | log | answer
status: stub | draft | stable
tags: [태그1, 태그2]
sources: [source-id, ...]        # sources/ 의 항목 id (해당 시)
created: YYYY-MM-DD
updated: YYYY-MM-DD
---
```

- `type: entity` — 고유명사(사람·조직·모델·제품·논문 등) 페이지. **위키 상호연결의 중추.**
  PARA 폴더는 유지하되 엔티티는 주로 `01_Concepts/`에 두고 `type`로 구분함.
- `type: concept` — 개념·원리·이론의 정의 페이지. 검수 시 정의 정확성·상위/하위 개념 혼동을 봄(§7-B).
- `type: overview` — 한 주제를 **서술형으로 종합**한 페이지(목차인 MOC와 다름). §11 참고.
- `type: answer` — 질의로부터 위키에 **환원된** 페이지.
- `status` — `stub`(껍데기) → `draft`(초안) → `stable`(안정).

### 본문 구조
```markdown
# 제목

> 한 줄 요약 (TL;DR)

## 핵심 내용
- ...

## 세부 설명
- ...

## 관련 문서
- [[관련문서1]]
- [[관련문서2]]

## 참고
- 출처는 본문에 (출처: source-id)로 표기. 여기엔 보조 링크/URL.
```

### 문체
- **두괄식** — 결론·핵심을 먼저, 부연은 뒤에.
- **개조식** — 불릿 중심으로 간결하게.
- **음슴체** — "~이다/~한다" 대신 "~임/~함" 체.
- 한 문장은 짧게, 불필요한 수식어 제거. 표·리스트를 적극 활용.

### 표현 형식 선택 기준
내용 성격에 맞는 구조를 고름 (작성·검수 공통 적용). 검수는 §7-A에서 점검함.

| 형식 | 사용 기준 |
|------|-----------|
| **도표(표)** | 비교가 필요할 때 · 같은 기준으로 여러 항목을 정리할 때 · 조건과 결과를 연결할 때 · 상태값·데이터 필드·리소스 목록·예외 처리를 정의할 때 |
| **1단계 블릿** | 짧은 속성을 정리할 때 · "라벨: 내용"으로 끝나는 정보일 때 · 항목들이 서로 동등한 수준일 때 · 빠른 확인이 목적일 때 · 단순 목록·요약을 제시할 때 |
| **서브 블릿** | 상위 항목을 세부 설명할 때 · 조건·예외·예시·절차가 필요할 때 · 담당자별 작업이 갈릴 때 · 입력값·출력값·저장값을 함께 설명할 때 · 한 줄로 쓰면 너무 길어지는 규칙을 정리할 때 |
| **번호 목록** | 순서가 중요한 절차·플로우를 설명할 때 |

### Wikilink 규칙
- **모든 페이지는 관련 페이지를 최소 2개 `[[ ]]`로 링크함** (예외: 순수 색인/로그). 링크 부족은 lint 대상.
- 새 개념·고유명사가 등장하면 `[[개념명]]`으로 링크함.
- 링크 대상 문서가 아직 없어도 괜찮음 — 다음에 만들 stub 신호.
- 별칭은 `[[파일명|표시텍스트]]` 형식.

### 출처·인용 규칙
- 주장에는 출처를 명시함: `... (출처: source-id)`.
- 원본을 인용·요약한 페이지는 frontmatter `sources:`에 해당 id를 나열함.

### 모순(contradiction) 처리
- 기존 지식과 충돌하는 정보는 **삭제하지 말고** 블록으로 보존함:

```markdown
> ⚠️ 모순: A는 X라 함 (출처: src-1) / B는 ¬X라 함 (출처: src-2). 미해결.
```

## 5. 수집 워크플로 (Ingest)

새 소스를 `sources/`에 추가하고 `sources/README.md` 표에 등록한 뒤:

1. 소스를 읽고 핵심 엔티티/개념/주장을 추출함. (필요 시 사용자와 강조점을 먼저 협의)
2. 관련된 기존 페이지를 **모두** 갱신함 — 한 소스가 보통 여러(목표 수준 ~10개 내외) 페이지를
   연쇄 변경함: 엔티티(`type:entity`)·개념(`type:concept`)·해당 주제의 종합(`type:overview`).
   없으면 적절한 PARA 폴더에 새 페이지(처음엔 `status: stub` 가능)를 만듦.
3. 새/변경 페이지를 `[[ ]]`로 상호 연결하고(페이지당 최소 2링크) 출처를 표기함.
4. 기존 지식과 충돌하면 모순 블록으로 보존함.
5. `00_Index/`의 MOC에 신규 페이지를 등록함.
6. `05_Logs/log.md`에 한 줄 기록을 append 함.
7. 파급 큰 문서(엔티티·concept·overview·answer·reference)는 `stable` 승격 전 **§7 검수**를 거침.

## 6. 질의 워크플로 (Query)

1. `00_Index/` MOC에서 관련 페이지를 찾음.
2. 해당 페이지(및 `[[링크]]` 이웃)를 읽고 **인용과 함께** 답을 합성함.
3. 위키에 없던 새 합성·탐색 결과는 `type: answer` 페이지로 **환원**해
   적절한 폴더에 두고 MOC·log에 등록함 (탐색이 휘발되지 않게 함).

## 7. 검수 워크플로 (Review) — 환각·오류 저감

원본을 가공한 **파급 큰 문서**는 작성한 모델이 아닌 **독립 모델(Codex)**이 교차검수함.
작성자(Claude)와 검수자(Codex)를 분리해 환각·과장·오귀속을 잡음.

**필수 검수 시점**: `status`를 `stable`로 승격하기 **직전**(일원화 기준). 대상은 `type`가
`entity` · `concept` · `overview` · `answer` · `reference`인 문서. 단순 `stub` · `log` · 색인은 제외함.
`draft` 단계도 요청 시 검수 가능(선택).
**검수자 호출**: `codex:rescue` 스킬 또는 `codex` 계열 에이전트로 Codex에 위임함.

**원본 전달은 필수**: 검수자에게 `[초안 + 인용한 원본(sources/ 파일 경로 또는 발췌)]`을 반드시 함께 넘김.
원본에 접근할 수 없으면 판정은 `검수 불가`이고, 그 문서는 **`stable`로 승격하지 않음**(`draft` 유지).

**3단계 — 작성(Claude) → 검수(Codex) → 수정(Claude)**

1. **작성 (Claude)** — 원본을 읽고 페이지를 컴파일함. 모든 주장에 `(출처: source-id)`를 태깅함.
   근거를 못 찾은 주장은 **지어내지 말고** ⚠️로 표시함.
2. **검수 (Codex)** — `[초안 + 원본]`을 함께 받아 주장별로 대조함. 아래 체크리스트로 판정함:
   `근거있음` / `과장` / `미근거(환각)` / `모순` / `검수 불가`. 판정과 근거를 구조화해 반환함.
3. **수정 (Claude)** — 플래그된 주장을 수정·삭제·재근거함. 미해결 충돌은 모순 블록(§4)으로 보존함.
   결과를 log에 `REVIEW`로 기록함.

**`stable` 승격 차단 조건** — 아래를 **모두** 만족해야 `stable`로 올림. 하나라도 어기면 `draft`에 머묾.
- **형식(A)·의미(B) 두 블록을 모두 통과**해야 함 (형식만 통과한 문서는 승격 불가).
- **캐치올**: A1~A4 · B1~B6 중 **하나라도** 실패·보류·미기재면 승격 불가
  (B4 비약·B5 불충분·B6 무관링크처럼 판정값에 없는 위반도 이 조항으로 차단됨).
- `미근거(환각)` · `모순` · `누락` · `검수 불가` 지적이 **0건**.
- `과장` 지적은 수정했거나 명시적 약화 표현(예: "~로 보임", 출처 한정)으로 처리됨.
- 수정으로 본문이 크게 바뀌면 **수정분을 재검수**함(Codex가 `재검수: 통과/보류`를 회신).

**Codex 검수 체크리스트 — 형식·의미 2블록 분리** (의미 검수가 형식 검수에 흡수되는 걸 막음. A·B를 **각각** 판정함)

**A. 형식 검수** (provenance·구조)
1. 출처 없이 단정한 문장이 있는가?
2. `sources:` 프론트매터와 본문 인용이 일치하는가?
3. `[[링크]]`가 실재 문서를 가리키고 최소 2링크를 충족하는가?
4. 표현 형식이 내용에 맞는가? (§4 표현 형식 선택 기준 — 비교·여러 항목 정리는 표, 순서 절차는 번호 목록 등)

**B. 의미 검수** (내용의 진위·정합 — 문자열 일치가 아니라 **뜻**을 봄)
1. 각 주장이 원본의 **뜻**과 맞는가? (맥락·조건·인과방향 포함. 키워드만 일치하는 건 통과 아님)
2. 원본보다 **세게 단정**(과장·일반화)하지 않았는가?
3. 원본의 핵심 단서·한계·반대증거를 **누락**해 오도하지 않는가?
4. 문서 내부 논리가 정합한가? (주장끼리 모순·비약 없음)
5. (answer·overview) 질문/주제에 **실제로·충분히** 답하는가?
6. `[[링크]]`가 **의미상 타당**한가? (개수 채우기용 무관 링크 금지)

판정값: `근거있음` / `과장` / `미근거(환각)` / `모순` / `누락` / `검수 불가`.
B의 6개 항목은 **각각** 판정·근거(원문 의미 요약 포함)를 기재함. 한 항목이라도 미기재면 **B 미통과**로 간주함
(B를 "전반적으로 근거있음"으로 뭉뚱그려 통과시키는 것 금지).

**검수 강도 (문서 중요도별 차등)** — 모든 검수를 전수로 하지 않고 깊이를 조절함.
- **경량** — 링크 · `sources:` 일치 · 출처 없는 단정 · 명백한 과장만 점검. (작은 draft, 저파급 문서)
- **표준** — 핵심 주장 전체를 source-id 원본과 대조. (검수 대상 기본값)
- **고강도** — 주장 단위 전수 대조 + 수정분 재검수. (`stable` 승격, overview, reference 요약, 외부 공개용)
- **`stable` 승격엔 경량 불가** — 의미 블록(B) 전체를 점검하는 표준·고강도만 인정함.

**합성/answer 페이지 주의**: 단일 원본이 없어 환각 위험이 큼. 모델 일반지식에서 나온 문장은
`(미검증: 모델 지식)`으로 명시함. 단 이 표시가 남아 있는 문서는 **`stable`로 승격하지 않음**(`draft` 유지) —
새 source를 찾아 근거를 보강한 뒤 승격함.

## 8. 유지보수 규칙 (Lint)

- **주기**: 2~4주마다 정기 점검함(수집이 뜸해도 멈추지 않음). 수집 때 발견한 문제도 즉시 보강함.
- MOC(00_Index)와 실제 페이지는 항상 일치시킴 (추가/삭제 시 동시 갱신).
- 끊어진 `[[링크]]`, 고아 페이지, **관련 링크 2개 미만 페이지**, 오래된 `stub`, 폐기·중복된 주장을 점검·보강함.
- 누락된 엔티티/개념 페이지, 연구 갭(다음에 채울 주제)을 식별함.
- `updated` 날짜를 갱신함. 점검 결과는 `LINT`로 log에 기록함.

## 9. 로그 (`05_Logs/log.md`)

**append-only.** 최신 항목을 **아래**에 추가함. 과거 줄은 수정·삭제하지 않음.
한 줄 포맷 (grep/Unix 도구 파싱 친화적):

```
YYYY-MM-DD | <작업유형> | <대상> | <한 줄 설명>
```

**작업유형**
- `INGEST` — 새 소스를 읽고 위키에 통합
- `QUERY` — 위키에 질문하고 답변 생성 (환원 포함 시 명시)
- `CREATE` — 새 문서 생성
- `UPDATE` — 기존 문서 수정
- `REVIEW` — Codex 교차검수 결과 반영 (§7)
- `LINT` — 위키 상태 점검 (모순·고아·누락 링크)
- `MAINT` — 구조·스키마 등 기타 정비

규칙: 작업이 끝날 때마다 반드시 기록함. log.md가 없으면 먼저 생성함.

## 10. 하지 말 것

- 사용자가 요청하지 않은 문서를 임의로 삭제하지 않음.
- `sources/`의 원본을 수정하지 않음 (읽기 전용).
- 확실하지 않은 정보를 사실처럼 적지 않음 — 불확실하면 명시하거나 모순 블록 사용.
- 출처 없는 주장을 단정하지 않음.
- 영어로 본문을 작성하지 않음 (기술 용어 병기는 허용).
- `assets/` 외 위치에 바이너리 파일을 두지 않음.
- log.md 기록을 생략하지 않음. 과거 로그를 고쳐 쓰지 않음.

## 11. Overview(종합) 페이지

- **정의**: 한 주제의 현재 이해를 **서술형으로 합성**한 페이지(`type: overview`). 단순 링크 목록인 MOC와 구분됨.
- **역할**: 흩어진 엔티티·개념 페이지를 하나의 이야기로 엮어 "지금까지 아는 것"을 보여줌.
- **위치**: 주제가 속한 PARA 폴더(주로 `01_Concepts/`)에 둠. 파일명 예: `LLM 개요.md`.
- **유지**: 관련 엔티티/개념이 갱신되면 함께 갱신함. 본문에서 구성 페이지를 `[[ ]]`로 링크하고 MOC에 등록함.
- **MOC와의 관계**: MOC(00_Index)는 "어디에 무엇이 있는가"(색인), Overview는 "그래서 무엇을 아는가"(종합).
